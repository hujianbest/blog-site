# ADR-0001: 选择模块化单体架构

## Status
Accepted (2026-05-08)

## Context
需要构建个人写作网站系统，包含以下功能模块：
- 内容管理（文章写作、编辑、发布）
- 多平台发布（转发到知乎、X等）
- 互动系统（评论、通知）

系统面临以下约束和考虑因素：
- 个人项目，团队规模小（solo开发者）
- 初期流量预期较低，但需要具备扩展能力
- 开发和维护成本需要控制在合理范围
- 需要支持未来可能的业务演进

## Decision
采用**模块化单体架构**而非微服务架构或传统三层单体架构。

### 架构特征
- **单一部署单元**：所有模块在同一应用进程中运行
- **模块化设计**：按业务领域划分清晰的模块边界
- **依赖方向**：模块间通过接口通信，依赖方向单向
- **数据隔离**：每个模块有自己的数据访问层，共享数据库

## Consequences

### 正面影响
1. **降低运维复杂度**
   - 无需管理多个服务的部署和监控
   - 简化日志聚合和问题诊断
   - 降低基础设施成本

2. **简化开发和测试**
   - 统一的代码库和IDE配置
   - 简化调试和单元测试
   - 减少服务间的集成测试复杂度

3. **性能优势**
   - 模块间通信无网络开销
   - 事务管理简单（ACID）
   - 低延迟的模块间调用

4. **演进路径清晰**
   - 模块边界清晰，未来可按需拆分为微服务
   - 技术栈统一，迁移成本低
   - 风险可控的渐进式重构

### 负面影响
1. **扩展性限制**
   - 水平扩展需要整体复制，无法单独扩展某个模块
   - 但对个人站点场景，这不是问题

2. **技术栈绑定**
   - 所有模块使用相同技术栈
   - 但对个人项目，这是优点而非缺点

3. **部署耦合**
   - 一个模块的错误可能影响整体
   - 通过良好的模块化设计和错误隔离可以缓解

## Alternatives Considered

### 选项A：微服务架构
**优点**：独立扩展、技术栈灵活、故障隔离
**缺点**：运维复杂度高、分布式事务、调试困难
**不选原因**：对于solo个人项目和预期流量，过度设计

### 选项B：传统三层单体架构
**优点**：实现简单、易于理解
**缺点**：模块边界不清、难以维护、扩展性差
**不选原因**：无法支撑业务复杂度，缺乏演进能力

## Implementation Notes

### 模块划分
基于DDD的Bounded Context：
- **Content Management Module**：内容管理
- **Publication Module**：多平台发布
- **Engagement Module**：互动系统

### 依赖规则
- 模块间依赖必须单向（上层依赖下层）
- 禁止循环依赖
- 通过接口（Repository/Service）通信

### 未来演进路径
如果某个模块需要独立扩展：
1. 提取该模块为独立服务
2. 通过API Gateway暴露接口
3. 使用消息队列处理异步通信
